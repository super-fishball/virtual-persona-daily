# feature/ — 跨子项目需求的整体维度

> 每个**跨子项目的完整需求（feature）**在这里有一个文件夹，承载子项目之上的整体内容。
> **单子项目内的需求不需要 feature 层**，直接在子项目内做。

## 命名

`feature/YYYY-MM-DD-<名>/`，例：`feature/2026-06-04-login/`

## 每个 feature 文件夹

| 文件 | 内容 |
|---|---|
| `spec.md` | 总 spec（整体要做什么、给谁、涉及哪些子项目）+ 设计理由（why） |
| `split.md` | 拆分索引（拆成哪些子项目任务，链到各自 spec/plan）+ 指向权威契约的指针 |
| `acceptance.md` | 端到端验收（整体怎么算跑通） |

## 契约形式（重要）

- 接口"是什么"（路由 / 请求响应 / 错误码）**只有一份权威源 = 后端拥有的机读 schema**（OpenAPI / JSON Schema / protobuf / 共享类型，格式随技术栈）。
- **feature/ 不复述接口，只在 `split.md` 里放指针**指向那份 schema。
- 防漂移靠**契约测试**：后端验证实现符合 schema，前端验证按 schema 调用。
- **设计理由（why）**放 `spec.md`，不放接口细节。
- 早期若还没 schema，散文接口约定**只作一次性脚手架**（标注"临时，schema 就位后删除"），不常驻。

## 归属与定稿

- feature 总 spec / 契约**无单一主人**，由涉及的子项目开发者集体共有。
- AI（brainstorming）起草 → 涉及方集体评审定稿 → 契约僵持时后端（提供方）兜底。
