## 2. 目标

实现契约③ 端点 `POST /v1/complete`：作为**薄 LLM 代理**——持有 LLM key、调单家 DeepSeek、协议适配、超时/重试——并对 LLM 输出施加**第三层通用横切 guardrail（block-only）**后返回。

**硬约束（守"薄"）**：只做代理与横切（凭证 / 协议 / 稳定性 / 安全过滤），**不承载事件生成等业务逻辑、不解析业务字段、不读业务语义**（CLAUDE.md §1、总 spec §6.4、forbidden #3）。

## 3. 范围

### 3.1 做（A1 切片）

| # | 项 | 依据 |
|---|---|---|
| 1 | **固定机械映射**：`systemInstruction→系统角色 / personality(数据槽)→数据消息 / realTime→上下文`，写死槽位、**不按内容做分支判断** | §6.4 · GW-1#2 |
| 2 | DeepSeek 协议适配（调上游补全 API） | CLAUDE.md §4 |
| 3 | LLM key 经 **env 注入**持有，不硬编码 / 不提交 | §5「LLM key」· forbidden #2 |
| 4 | 请求级**超时 + 有限重试**，错误向上游透传 | §3.4 · 本地 high-impact |
| 5 | **第三层输出 guardrail**：三类各起步规则，命中 → **422 block-only** | §5.1 ③ · §6.4 · GW-1#1 |
| 6 | 错误透传：guardrail → 422；上游不可用/超时 → 502 | 契约③ responses |

### 3.2 不做（守边界，spec 全文不依赖）

- 多家 provider / fallback、预算上限 / 精细限流（§3.4 · forbidden #4）。
- **静默改写 / 脱敏**——guardrail 是 block-only，命中要么放行要么 422，不在网关"修"内容（GW-1#1：改写 = 篡改业务产出）。
- **按内容做分支判断 / 解析业务字段 / 读业务语义**（如判断"事件是否连贯"）——越此即非本层（GW-1 硬线）。
- 注入检测模型 / PII 脱敏 / 对抗测试（总 spec §5.1「最小必要」，属后续加深）。

> 越界即停：命中"网关堆业务逻辑 / 扩多家 LLM 抽象 / 单方改既有契约"任一 → 暂停，回根 §6 / 走 feature 层（scope.md「越界处理」）。

## 4. 接口 / 数据结构（引用契约③，不复述字段）

- **权威源**：`POST /v1/complete` 的请求 / 响应 / 错误码以 [`../contracts/gen-aigw.openapi.yaml`](../contracts/gen-aigw.openapi.yaml) 为准；本 spec **不复述字段定义**，只记映射与协议适配要点。
- **机械映射**（契约③ 三分立字段 → DeepSeek 调用，写死槽位）：

  | 契约③ 字段 | 角色（DeepSeek 调用内） | 约束 |
  |---|---|---|
  | `systemInstruction` | 系统指令 / system 消息 | 指令文本，不含用户数据 |
  | `personality`（数据槽） | **数据 / 用户消息**，**不并入指令文本** | 与 systemInstruction 物理分离（GW-2 可审计） |
  | `realTime` | 生成上下文（现实时间锚点） | — |

- **为何分立字段而非 `messages[]`**（GW-2）：分立字段把"性格当数据 vs 指令"的**物理分离留在契约层、可见可审计**；代价 = gateway 做一次**固定槽位映射**，由 GW-1#2 钉死为"机械、不按内容判断"，故**不构成业务 prompt 拼装**（拼装语义归 gen，总 spec §5.1）。
- **协议适配**：仅做契约③ ↔ DeepSeek 的协议/字段形状转换；不注入业务规则。

## 5. 横切策略

### 5.1 凭证

- `DEEPSEEK_API_KEY` 经**环境变量 / 密钥管理注入**（见 [`../.env.example`](../.env.example)），**代码 / 日志 / 仓库均不硬编码、不提交**（forbidden #2）。
- **日志不输出完整 prompt / 响应**（含 systemInstruction / personality / 生成文本），防隐私或敏感数据落日志（forbidden #5、#9）。

### 5.2 超时 / 重试（错误透传）

- 请求级**超时**；瞬时失败**有限次重试**，重试耗尽 → **502** `{code: upstream_unavailable}`。
- **超时阈值 / 重试次数留 ④ plan 定**（实现层，合 F4·G-1 裁决）。
- **已知取舍（记录）**：A1 **重试非幂等**——gen↔gateway 这一程的重试会**重复消耗 DeepSeek 配额**。属首刀刻意取舍，幂等键属后续刀（总 spec [§10 契约演化候选](../../../feature/2026-06-10-virtual-persona/spec.md) · F4·G-1）。

### 5.3 第三层输出 guardrail（三类各起步规则 · block-only · 不解析业务字段）

对 LLM **输出文本**做通用横切规则匹配；**只看输出文本、不读业务语义**。

| 类 | A1 起步规则 | 命中 |
|---|---|---|
| 注入回显 | denylist：输出回显常见注入字面量（如"忽略以上指令"类） | 422 |
| 泄漏 prompt | 检测输出是否回显 `systemInstruction` 片段 / 特征 | 422 |
| 不安全内容 | 起步 denylist 词表 | 422 |

- **block-only**（GW-1#1）：命中即 **422** `{code: guardrail_blocked}`，**不做静默改写 / 脱敏**。
- **不解析业务字段、不读业务语义**（GW-1 硬线）：一旦需要理解业务（如判断"事件是否连贯"），即**非本层** → 暂停回根 §6，不在 ai-gateway 实现。
- "三类各起步规则" = 首刀即实例化 guardrail 层的**完整三档骨架**，后续刀只加厚规则、不改结构（不引注入检测模型，§3.2）。

### 5.4 错误码（与契约③ Error 一致）

| 场景 | HTTP | `code`（机读） |
|---|---|---|
| guardrail 命中（注入回显 / 泄漏 prompt / 不安全内容） | 422 | `guardrail_blocked` |
| 上游 DeepSeek 不可用 / 超时（重试后仍失败） | 502 | `upstream_unavailable` |

## 6. 验收标准（A1 · 走薄）

### 6.1 那一条契约测试（硬门 · ④ TDD · 提供方侧）

- **场景**：合法 `CompletionRequest` + **mock DeepSeek**（不走真实上游、不耗 key）→ 返回 **200** `CompletionResponse{text}`，**符合契约③ schema**。
- **额外断言（机械映射可验）**：传给 DeepSeek 的 **system 消息内容 == `systemInstruction`**；`personality` 进**数据 / 用户消息**、**未并入指令文本**（GW-2 物理分离可审计）。
- **门禁口径**：此为首刀"那一条契约测试"，是 A1 spec 的硬验收门。

### 6.2 实现自检（伴随实现，不计入 6.1 那一条契约测试）

- guardrail 三类规则各带 **≥1 条 block→422 单测**（注入回显 / 泄漏 prompt / 不安全内容）。
- 502 上游失败路径 ≥1 条单测（mock 上游不可用 → 502 `upstream_unavailable`）。
- 凭证不入日志、prompt/响应不入日志的自检（development-guard.md 本服务特有检查）。
- 四道门（[`../Makefile`](../Makefile)）：`make test / lint / typecheck / build` 绿。

> 422/502 的更全用例（多模式 denylist、各类边界）排**后续刀**——首刀只保 6.1 一条契约测试穿 happy-path + 6.2 实现自检，合"走薄"。
